What is Git

Git is a version managing system created by Linus Torvald, the same guy who also created Linux. It's a way to keep track of code changes and allows for easy collaboration on that code. It also serves as a way to back up your code, if you use some repository (=place where your code lives) that can be accessed online.

What is GitHub

GitHub runs git. It adds some functionality that git doesn't provide to improve sharing/backing-up/colaborating on code. GitHub is a collection store for everone who wants to share their code. There are some alternatives to GitHub, ie. Bitbucket, GitLab, ... They all serve a specific target audience, but GitHub offers free public repositories (in order to stimulate open source code), which is why we are using it.

GitHub's mascotte is Octocat. He will make all kinds of anouncements about the GitHub ecospace and will be your guide that let's you know what's up with your repositories

Checking the status of your git repository

git status

This command gives you an overview of what has happened in the repository since the last time you commited some changes. It will list newly added, untracked files and/or files that have been changed

Staging files

Staging means that git needs to keep track of this file. Sometimes it's necessary to exclude files from git, for instance because they contain sensitive information like passwords, see .gitignore

git add .

the . operator means 'all', this will add all the files to te stage. This means git will start keeping track of them. You will usually use this command.

Sometimes you want to add files individually, then you just change the . operator with the path to the filename:

git add css/main.css

Commiting

When you commit, it means that you have created a new version of your application. You should commit for every functionality you add. This seems like a lot of work, but it will save your life when you need to changes that you have previously removed. There are no rules, but this is a best practice that what will make your life as a developer easier.

git commit -m "Add description to readme"

There is also a way to do an add and commit at the same time. Use the following shorthand:

git commit -am "Add description to readme"

The a flag stands for add, the m-flag for message

A lot is written about when to commit and what commit messages to use. I'll save you the details, but try to use a descriptive name for your commit messages, so that by looking at the commit history it is clear what changes have been made. Messages like 'Change the document' are pretty useless, because they are too generic. 'Add description chapter to readme' is a lot more clear. Usually commit messages also start with a verb (add, change, remove, ...)

Discarding changes

It's possible that you made some changes to a file that you want to revert back to the original, meaning the last time you have committed that file. If you wan't to discared your changes you can do this by discarding the file:

git checkout -- css/main.css

If you want to discard all changes, you use the same operator as described before:

git checkout -- .

Branches

Branches are versions of your application. The main branch is usually called main or master. You can see all the branches of your repository by executing the following command:

git branch

If you want to create a new version, to add a new feature, or because you want to experiment with your code, but don't want to mess up your original code, you create a new branch

git checkout -b experimental-feature

A new branch experimental-feature has been created and you are switched to this branch automatically. To see this, just execute the git branch command again.

If you want to switch to a different branch, you use the following command

git checkout -b master

Note that you can only do this if you have committed the changes that in your branch. You can chose to either commit, or discard your changes as described above. This is as a failsafe to make sure you don't lose your changes by accident.

If you want to remove a branch you can execute the following command:

git branch -d experimental-feature

Merging branches

At some point, when you are done working on changes in a specific branch, you want to merge that branch with the official version, which in your case will be the main/master branch

You can merge this by executing the following command:

git merge master experimental-feature

Now it's possible that the files in your main branch have changes in the meanwhile. Then you will have caused a so called merge conflict. You will have to resolve these conflicts manually. Git has a special way to show you where the conflict takes place. After you have fixed the merge conflict, you can now commit these changes with a commit messages that says something like fix merge conflicts.

Pushing your changes to your online repository

Now that you have made some changes to your repository, you might want to back them up or publish them to the rest of the world. You do this by pushing your changes to the origin. Execute the following command:

git push origin master

Some standard Git files

.gitignore

This file lists all the files that should never be tracked by git. If they are listed in this file, Git simply doesn't know they exist. This is handy if you want to store passwords, but you don't want them to be committed to the repository. They could end up online somewhere and you might not want that...

README.md

This file is usually used as an 'about me' of the repository. It explains what the application does, which version, what's on the changelog. Basically anything can go there, I use it to keep track of TODO's as well. It can be anything you want, but don't forget that this file is the first thing people will see when they visit your repository online, so don't do anything stupid like storing passwords or sensitive data in there.

As you might have noticed, it has a weird file extension: .md .md stands for markdown and is basically a simple form of HTML. You can check out the syntax here: Markdown cheat sheet.
